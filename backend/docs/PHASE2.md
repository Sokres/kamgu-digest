# Фаза 2 (отложено): очередь и расписание

Когда станет известна **частота** и ожидаемая **нагрузка**, имеет смысл вынести генерацию дайджеста из синхронного `POST /digests` в асинхронный контур.

## Рекомендуемые шаги

1. **Очередь задач** — Cloud Tasks (GCP), SQS (AWS), или Redis + RQ/Celery на Fly.io.
2. **Эндпоинты** — `POST /digests/jobs` → `202` + `job_id`; `GET /digests/jobs/{id}` → статус и результат.
3. **Воркер** — тот же Docker-образ, отдельная команда `python -m worker` (или второй процесс), читает очередь и вызывает `run_digest`.
4. **Планировщик** — Cloud Scheduler / EventBridge / cron на платформе, дергает HTTP или кладёт задачу в очередь по расписанию.
5. **Доставка** — webhook или email при `completed` (опционально).

MVP остаётся полезным для ручных запусков и низкой нагрузки; фаза 2 не ломает контракт API, а расширяет его.
