# Техническое задание  
## Сервис автоматической генерации дайджестов по научным публикациям (ИИ-агент)

**Версия документа:** 1.0  
**Назначение:** описание работ по разработке и внедрению сервиса для заказчика (формулировки в настоящем/будущем времени — план работ).

---

## 1. Цель и задачи

### 1.1. Цель  
Создать облачный сервис, который по заданным тематическим запросам (на русском и/или английском языке) будет **собирать актуальные открытые научные публикации** из международных источников, **отбирать наиболее релевантные** материалы и **формировать текстовый дайджест** с помощью большой языковой модели (LLM). Дайджест будет выдаваться **на русском и английском языках**, чтобы его можно было использовать в отчётности и рассылках.

### 1.2. Задачи  
- обеспечить **периодическое** (ориентир — **не реже одного раза в квартал** на профиль/направление) автоматическое обновление обзора литературы;  
- дать возможность **разовых ручных запусков** по запросу (например, перед конференцией или отчётом);  
- минимизировать дублирование одной и той же работы из разных баз;  
- заложить возможность **сравнения с предыдущим периодом** (какие статьи «вошли» в топ, как изменилась цитируемость внутри выбранной выборки, сдвиги тематических акцентов);  
- подготовить сервис к **развёртыванию в облаке** (контейнер, переменные окружения, рекомендации по таймаутам и секретам).

---

## 2. Область применения и пользовательский сценарий

**Основной сценарий (квартальный):**  
заказчик задаёт **профиль** (идентификатор направления) и **тематические запросы**; по расписанию (внешний планировщик, например раз в три месяца) сервис запускает конвейер, сохраняет снимок топ-статей, сравнивает с предыдущим снимком и формирует **структурированное описание изменений** плюс **текст дайджеста** с оговоркой о том, что выводы носят справочный характер.

**Дополнительный сценарий:**  
разовый запрос дайджеста без обязательного сравнения с прошлым периодом — для оперативных нужд.

---

## 3. Источники данных

Планируется использование **открытых программных интерфейсов** (без закупки платных баз на стороне разработки):

| Источник | Назначение |
|----------|------------|
| **OpenAlex** | широкий охват научных работ, метаданные, концепты, цитирование |
| **Semantic Scholar** | дополнение выборки, абстракты и метаданные |

Для корректной работы с API планируется соблюдение рекомендаций источников (в т.ч. указание контактного e-mail в User-Agent для OpenAlex при развёртывании).

---

## 4. Функциональные требования

### 4.1. Сбор и отбор публикаций  
- Поиск кандидатов по объединённой тематической строке (из нескольких фраз запроса).  
- Ограничение по годам публикации — по запросу заказчика.  
- **Дедупликация** записей, пришедших из разных источников (по DOI, идентификаторам и т.п.).  
- **Ранжирование** кандидатов по релевантности к запросам.  
- Настраиваемые лимиты: число кандидатов на входе и число статей, передаваемых в LLM для итогового текста.

### 4.2. Генерация дайджеста (LLM)  
- Формирование **дайджеста на русском** и **дайджеста на английском** на основе отобранных карточек публикаций (название, год, краткое содержание/аннотация при наличии).  
- В ответе — список использованных публикаций и метаданные выполнения (время, предупреждения при частичных сбоях источников).

### 4.3. Периодический режим («квартальный» обзор)  
- Хранение **снимка** топ-публикаций по профилю в **локальной базе (SQLite)** на стороне сервиса (файл на постоянном томе при деплое).  
- При очередном запуске — **сравнение** с предыдущим снимком: прирост цитирований в рамках выборки, вход/выход из топ-K, изменения по тематическим концептам (OpenAlex).  
- Генерация **текстового блока** от LLM с учётом структурированных отличий и **дисклеймера**.  
- Защита служебного вызова по расписанию: опциональная проверка секретного ключа в заголовке HTTP-запроса.

### 4.4. Программный интерфейс (API)  
- REST API (планируется фреймворк **FastAPI**), документация **OpenAPI/Swagger** на эндпоинте `/docs`.  
- Эндпоинт проверки работоспособности: `/health`.  
- Эндпоинт разового дайджеста: например `POST /digests`.  
- Эндпоинт периодического режима со снимками: например `POST /digests/monthly` (название эндпоинта может отражать «месячный» шаг хранения; **частота запусков задаётся планировщиком** — в т.ч. **раз в квартал**).

### 4.5. Надёжность  
- Повторные попытки при временных ошибках и лимитах API источников и LLM (настраиваемые паузы и число ретраев).  
- Логирование для диагностики.

---

## 5. Нефункциональные требования

- **Развёртывание:** Docker-образ, переменная порта для PaaS (Cloud Run, Fly.io и аналоги).  
- **Секреты:** ключ API LLM хранится в менеджере секретов окружения, не в репозитории.  
- **Производительность:** допускаются длительные запросы (рекомендуется таймаут сервиса 5–15 минут в зависимости от объёма).  
- **Тестирование:** автоматические тесты на ключевые модули (дедупликация, модели данных, логика сравнения снимков).

---

## 6. Ограничения и допущения

- Качество и полнота выборки зависят от **открытых API** и лимитов поставщиков.  
- **Квартальная** периодичность реализуется **расписанием внешнего планировщика** (cron, Cloud Scheduler и т.д.), обращающегося к HTTP API сервиса с нужным интервалом (например, каждые 90 дней).  
- Стоимость в основном определяется **объёмом вызовов LLM**; параметры `top_n` и модель можно согласовать с заказчиком для баланса цена/качество.

## 7. Этапы и результаты работ (план)

| Этап | Содержание | Результат |
|------|------------|-----------|
| 1 | Реализация конвейера: источники → дедупликация → скоринг → LLM, API разового дайджеста | Работающий сервис и документация API |
| 2 | Режим снимков, сравнение периодов, защита служебного вызова | Квартальный/периодический сценарий по расписанию |
| 3 | Контейнеризация, рекомендации по деплою, настройка секретов и тома для БД снимков | Готовность к промышленному запуску |

*При необходимости отдельным этапом может быть запланировано развитие: **очередь задач**, асинхронные job’ы и доставка результата по e-mail/webhook — без изменения базового сценария.*
