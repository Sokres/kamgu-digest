# Техническое задание (предложение для согласования)

## Сервис автоматической генерации дайджестов по научным публикациям (ИИ-агент)

| | |
|---|---|
| **Версия** | 1.1 (предложение исполнителя) |
| **Назначение** | Зафиксировать **планируемый** объём и характер работ для заказчика. Заказчик не готовит ТЗ самостоятельно — настоящий документ выступает **проектом ТЗ**, подлежащим согласованию и при необходимости правкам до подписания договора или приложения к нему. |
| **Статус** | Работы по разработке **не начаты** / не завершены в объёме, описанном ниже; формулировки ниже описывают **намерение и обязательства сторон после согласования**, а не факт уже сданной системы. |

---

## 1. Цель и задачи

### 1.1. Цель

По согласованию с заказчиком разработать и внедрить облачный сервис, который по заданным тематическим запросам (на русском и/или английском языке) будет:

- собирать **актуальные открытые** научные публикации из международных источников с публичными API;
- отбирать **наиболее релевантные** материалы к запросам заказчика;
- формировать **текстовый дайджест** с помощью большой языковой модели (LLM);
- выдавать дайджест **на русском и на английском языках** для использования в отчётности, обзорах литературы и рассылках.

### 1.2. Задачи (планируемые к выполнению в рамках согласованного объёма)

- обеспечить возможность **периодического** обновления обзора литературы — ориентир по частоте: **не реже одного раза в квартал** на профиль/направление (точная периодичность настраивается **внешним планировщиком**, обращающимся к сервису);
- предусмотреть **разовые ручные запускы** по запросу (например, перед конференцией или отчётным периодом);
- заложить **дедупликацию** записей из разных баз, чтобы одна и та же работа не учитывалась многократно;
- предусмотреть **сравнение с предыдущим периодом**: какие статьи вошли в топ, как меняется цитируемость внутри выбранной выборки, сдвиги тематических акцентов;
- подготовить решение к **развёртыванию в облаке или на сервере заказчика**: контейнеризация, переменные окружения, рекомендации по таймаутам, секретам и резервному копированию данных снимков.

---

## 2. Область применения и пользовательский сценарий

**Основной сценарий (периодический обзор, в т.ч. квартальный):**  
заказчик задаёт **профиль** (идентификатор направления) и **тематические запросы**; по расписанию внешний планировщик инициирует запуск конвейера; сервис сохраняет снимок топ-статей, сравнивает с предыдущим снимком и формирует **структурированное описание изменений** и **текст дайджеста** с оговоркой о справочном характере выводов (дисклеймер).

**Дополнительный сценарий:**  
разовый запрос дайджеста **без** обязательного сравнения с прошлым периодом — для оперативных задач.

Границы сценариев и приоритет функций уточняются при согласовании настоящего документа.

## 3. Источники данных

Предполагается опора на **открытые программные интерфейсы** (без закупки платных научных баз на стороне исполнителя в базовом объёме), в частности:

| Источник | Назначение |
|----------|------------|
| **OpenAlex** | широкий охват научных работ, метаданные, концепты, цитирование |
| **Semantic Scholar** | дополнение выборки, абстракты и метаданные (при включении в конфигурации и соблюдении лимитов API) |

Для корректной работы с API планируется соблюдение правил источников (в т.ч. указание контактного e-mail в User-Agent для OpenAlex — по рекомендациям поставщика).

*Примечание:* расширение списка источников или подключение платных баз — **отдельное согласование** объёма и стоимости.

## 4. Функциональные требования (к разрабатываемой системе)

### 4.1. Сбор и отбор публикаций

- Поиск кандидатов по объединённой тематической строке (из нескольких фраз запроса).
- Ограничение по годам публикации — по запросу заказчика.
- **Дедупликация** записей из разных источников (по DOI, идентификаторам и т.п.).
- **Ранжирование** кандидатов по релевантности к запросам.
- Настраиваемые лимиты: число кандидатов на входе и число статей, передаваемых в LLM для итогового текста.

### 4.2. Генерация дайджеста (LLM)

- Формирование **дайджеста на русском** и **дайджеста на английском** на основе отобранных карточек публикаций (название, год, краткое содержание/аннотация при наличии).
- В ответе — перечень использованных публикаций и служебные метаданные выполнения (время, предупреждения при частичных сбоях источников).

### 4.3. Периодический режим («снимки» и сравнение периодов)

- Хранение **снимка** топ-публикаций по профилю в **согласованной с заказчиком базе данных** на стороне сервиса (например, PostgreSQL в продакшене или упрощённый вариант для пилота — по договорённости).
- При очередном запуске — **сравнение** с предыдущим снимком: прирост цитирований в рамках выборки, вход/выход из топ-K, изменения по тематическим концептам (OpenAlex), где применимо.
- Генерация **текстового блока** от LLM с учётом структурированных отличий и **дисклеймера**.
- Защита служебного вызова по расписанию: опциональная проверка секретного ключа в заголовке HTTP-запроса.

### 4.4. Программный интерфейс (API)

- REST API; в предложении исполнителя — фреймворк **FastAPI** с документацией **OpenAPI/Swagger** на эндпоинте `/docs` (окончательный стек фиксируется в ходе согласования).
- Эндпоинт проверки работоспособности: `/health`.
- Эндпоинт разового дайджеста: например `POST /digests` (имена и тела запросов уточняются в спецификации API при сдаче).
- Эндпоинт периодического режима со снимками: например `POST /digests/monthly` — **частота запусков задаётся внешним планировщиком** (в т.ч. **раз в квартал**), а не только «ежемесячно»; наименование эндпоинта может быть скорректировано для ясности.

### 4.5. Надёжность

- Повторные попытки при временных ошибках и лимитах API источников и LLM (настраиваемые паузы и число повторов).
- Логирование для диагностики.

### 4.6. Возможное развитие (не входит в базовый объём без отдельного согласования)

- Очередь задач, асинхронные задания, доставка результата по e-mail или webhook.
- Дополнительные режимы поиска (например, веб-сниппеты через специализированные API) — при отдельном ТЗ или допсоглашении.

## 5. Нефункциональные требования

- **Развёртывание:** поставка в виде Docker-образа (или эквивалента), настраиваемый порт, при необходимости — рекомендации для PaaS (Cloud Run, Fly.io и аналоги).
- **Секреты:** ключ API LLM и прочие секреты — только в менеджере секретов окружения или защищённых файлах на сервере, **не** в репозитории исходного кода.
- **Производительность:** допускаются длительные запросы; рекомендуется закладывать таймаут на стороне прокси/клиента **5–15 минут** в зависимости от объёма (согласуется с инфраструктурой заказчика).
- **Тестирование:** автоматические тесты на ключевые модули (дедупликация, модели данных, логика сравнения снимков) — в объёме, достаточном для приёмки по договору.

## 6. Ограничения и допущения

- Качество и полнота выборки зависят от **открытых API** и лимитов поставщиков; исполнитель не гарантирует исчерпывающий охват всех журналов мира.
- **Квартальная** (или иная) периодичность реализуется **расписанием внешнего планировщика** (cron, Cloud Scheduler и т.д.), вызывающего HTTP API сервиса с нужным интервалом.
- Эксплуатационные расходы (в первую очередь **вызовы LLM**) зависят от объёма запросов и выбранной модели; параметры `top_n`, модель и лимиты **согласуются** с заказчиком для баланса цена/качество.

| Этап | Содержание | Планируемый результат |
|------|------------|------------------------|
| 1 | Конвейер: источники → дедупликация → скоринг → LLM; API разового дайджеста | Работающий сервис и документация API |
